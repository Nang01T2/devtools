What is Next.js?

nextjs

What is Next.js?

Next.js is a React framework created by the company Vercel.

Framework vs Library

The main difference between a framework and a library is that a framework provides a predefined structure and you build your code following that structure.

For example, in Next.js (using the App Router), creating a file called about/page.js automatically becomes the /about route — because the framework is designed that way.

React, on the other hand, is a library, so you have much more freedom in how you structure and organize your code.

Main Advantages of Next.js

  1. SEO-friendly thanks to SSR (Server-Side Rendering)
  2. Great developer experience — routing and configuration are extremely convenient
  3. Excellent TypeScript support out of the box
  4. Easy deployment — especially seamless with Vercel
  5. Stability & performance — frequent updates and improvements

What is SSR (Server-Side Rendering)?

In the flow: Database → Server → Client

  • If the server generates the HTML → SSR
  • If the client (e.g. React) generates the HTML → CSR

Server-Side Rendering means the web page is rendered on the server before it is sent to the browser.
The server processes the request, renders the full HTML, and sends a completely formed page to the user's device.

In other words: the HTML content that will be displayed is pre-rendered on the server.

Benefits of SSR

  1. Faster initial load
    → If you set your network to Slow 3G and refresh a plain React (CSR) app, you often see a white screen for a long time.
    SSR usually delivers a fully rendered page quickly.

  2. SEO-friendly
    → Search engine crawlers can immediately read the content from the server response, making indexing much easier.

  3. Better resilience
    → Even with poor internet, disabled JavaScript, or no JavaScript support, users can still see content.

What is CSR (Client-Side Rendering)?

In Client-Side Rendering, rendering happens in the browser.

The server sends a minimal HTML page + JavaScript bundle.
(In a default unoptimized React app, this is usually just an empty <div id="root"></div> → white screen until JS loads and runs.)

Then the JavaScript executes, fetches data, and builds/updates the page in the browser.

Benefits of CSR

  1. High interactivity
    → The browser handles most rendering → enables very dynamic and responsive UIs.

  2. Lower server load
    → Server sends minimal data → cheaper hosting costs.

  3. Fast after initial load
    → Once JavaScript is loaded and the app is hydrated, navigation and updates feel very fast.

What is SEO (Search Engine Optimization)?

Search Engine Optimization (SEO) is the process of improving a website so that it ranks higher in search engine results pages (SERPs).
It is one of the most important online marketing strategies, helping users quickly find information through specific search terms or questions.

Server Components vs Client Components in Next.js

Rendering means converting JSX/TSX (or JavaScript) into actual HTML that the browser can display.

Next.js, by default, performs rendering on the server (Server Components), which is why it supports SSR out of the box.

How Next.js rendering works

  1. The server sends initial HTML (static or pre-rendered) to the browser first.
  2. Then the browser downloads JavaScript.
  3. React hydrates the page — attaching event listeners and making HTML elements interactive.

This process of turning static HTML into a fully interactive React application is called hydration.

In short:
First deliver basic HTML → later use JavaScript to make it interactive = hydration

Components marked with 'use client' are Client Components.

  • Only Client Components are hydrated (receive interactivity).
  • Server Components are rendered only on the server and do not get hydrated.
  • 'use client' does not mean “rendered on the client” — it means “this component needs interactivity / browser APIs”.

Even 'use client' components are rendered on the server first, then hydrated in the browser.

Metadata in Next.js

You can define metadata by exporting a metadata object from page.js or layout.js files (not from regular components).

Metadata from nested layouts/pages is merged automatically.

 javascript
export const metadata = {
  title: "This is the title",
  description: "This is the description"
}

Using title templates

 javascript
import { Metadata } from "next";

export const metadata: Metadata = {
  title: {
    template: "%s | My Site",
    default: "My Site",
  },
  description: "Default description",
}
  • Pages can fill in the %s part (e.g. "About | My Site")

Dynamic metadata (generateMetadata)

For dynamic routes like [id]/page.js, use the async generateMetadata function.

 javascript
async function getMovie(id: string) {
  const res = await fetch(`${URL}/${id}`);
  return res.json();
}

export async function generateMetadata({ params: { id } }: { params: { id: string } }) {
  const movie = await getMovie(id);
  return {
    title: movie.title,
  };
}

export default async function MovieDetailPage() {
  // page content...
}

Benefit:
Because data fetching happens on the server and generateMetadata can reuse the same fetch, there is no double fetching — very efficient.

Major Updates & Latest Features (Next.js 15 → 16 Series, 2025–2026)

Next.js 16 (October 2025) and 16.1 (December 2025) brought transformative changes:

1. Turbopack (Stable & Default)

  • Replaces webpack in most cases
  • Next.js 16.1: File system caching → dramatically faster next dev restarts & HMR
  • Incremental computation → scales to huge apps without slowdowns
  • Use: automatic in new projects; next dev --turbo or config if needed

2. Cache Components & Composable Caching (New Programming Model)

  • Mix static, cached, dynamic content in one route
  • Directives: 'use cache', 'use cache: private', 'use cache: remote'
  • APIs: cacheLife, cacheTag, updateTag(), refresh(), refined revalidateTag()/revalidatePath()
  • Enables instant navigations, no double fetches, predictable behavior

3. Partial Prerendering (PPR) — Stable & Recommended

  • Static shell at build time + dynamic holes streamed at request time
  • Best balance: static speed + dynamic data freshness
  • Works seamlessly with Cache Components

4. React Compiler (Stable, Built-in)

  • Automatic optimizations for Client Components
  • Reduces bundle size & runtime cost
  • Enable via experimental: { reactCompiler: true } (often default in new apps)

5. Enhanced Routing & Navigation

  • Leaner, faster page transitions
  • Optimized prefetching & layout deduplication
  • Better support for View Transitions (React 19.2)
  • New hooks & navigation improvements

6. Developer Tools & Debugging

  • Next.js Bundle Analyzer (experimental in 16.1) — visualize bundle sizes
  • Easier Node.js debugging: next dev --inspect
  • Redesigned error overlays & stack traces

7. React 19.2 Integration

  • View Transitions, useEffectEvent, <Activity/>, improved refs & hooks (use, useActionState)

8. Other Notable Additions

  • Build Adapters API (alpha) — customize build output for any runtime
  • Streaming metadata support
  • Typed routes & route helpers (improved TypeScript)
  • MCP Server support — real-time access for coding agents (advanced)

2026 Recommendation:

  • App Router only for new projects
  • Default to Server Components + PPR + Cache Components
  • Use Turbopack everywhere possible
  • Leverage React Compiler for Client Components
  • Adopt composable caching over legacy patterns

Code Snippets for Key Features (Next.js 16.1+)

1. Cache Components Examples

Basic 'use cache' in a Server Component (caches the entire component output):

 tsx
// app/dashboard/page.tsx
'use cache';  // Caches this route/component across requests

export default async function Dashboard() {
  const data = await fetch('https://api.example.com/stats', {
    cache: 'force-cache',  // or use next: { tags: ['stats'] }
  }).then(res => res.json());

  return (
    <div>
      <h1>Dashboard Stats</h1>
      <pre>{JSON.stringify(data, null, 2)}</pre>
    </div>
  );
}

Using cacheLife and cacheTag for fine-grained control:

 tsx
// app/products/[id]/page.tsx
import { cacheLife, cacheTag } from 'next/cache';

export default async function ProductPage({ params }: { params: { id: string } }) {
  cacheTag(`product-${params.id}`);           // Tag for invalidation
  cacheLife(3600);                            // Cache for 1 hour (or use { expire: 3600 })

  const product = await fetch(`https://api.example.com/products/${params.id}`, {
    next: { revalidate: 3600, tags: ['products'] }
  }).then(res => res.json());

  return <div>Product: {product.name}</div>;
}

On-demand revalidation in a Server Action:

 tsx
// app/actions.ts
'use server';
import { revalidateTag } from 'next/cache';

export async function updateProduct(id: string) {
  // ... update logic ...
  revalidateTag(`product-${id}`);
}

2. Partial Prerendering (PPR) Setup

PPR is now the default hybrid behavior with Cache Components (static shell + dynamic holes via Suspense).Example route with PPR (static parts cached, dynamic streamed):

 tsx
// app/blog/[slug]/page.tsx
import { Suspense } from 'react';

async function Comments({ slug }: { slug: string }) {
  const comments = await fetch(`https://api.example.com/comments/${slug}`).then(res => res.json());
  return <ul>{comments.map(c => <li key={c.id}>{c.text}</li>)}</ul>;
}

export default function BlogPost({ params }: { params: { slug: string } }) {
  return (
    <div>
      <h1>Blog Post: {params.slug}</h1>
      <p>This part is static (cached shell).</p>
      
      <Suspense fallback={<div>Loading comments...</div>}>
        {/* Dynamic hole – streamed in at request time */}
        <Comments slug={params.slug} />
      </Suspense>
    </div>
  );
}

3. Turbopack ConfigurationTurbopack is stable/default in dev & builds. Customize via next.config.js / next.config.ts:

 ts
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  // Enable Turbopack explicitly (often automatic in v16+)
  turbopack: {
    // Optional: customize rules, loaders, etc.
  },

  // Enable file system caching for faster dev restarts (Next.js 16.1+)
  experimental: {
    turbopackFileSystemCacheForDev: true,
  },

  // React Compiler (pairs well with Turbopack)
  experimental: {
    reactCompiler: true,
  },
};

export default nextConfig;

Run with: next dev --turbo (or just next dev in most new projects).
